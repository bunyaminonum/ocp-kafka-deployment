# Prod-Shaped Kafka (KRaft) Deployment — Custom SCC + TLS + SASL + GitOps

Bu klasör, ana [README](../README.md)'deki playground kurulumunun **güvenlik ve operasyon
açısından sertleştirilmiş** halidir. Playground'da bilerek atlanan/basitleştirilen üç şey
burada gerçek mekanizmalarla kuruldu ve **canlı testlerle kanıtlandı**:

1. **Custom SCC** (playground'daki `podSecurity.enabled=false` yerine)
2. **TLS + SASL/PLAIN** (playground'daki çıplak PLAINTEXT yerine)
3. **ArgoCD (pull-based GitOps)** (playground'daki GitHub Actions/push-based yerine —
   ikisi birbirini geçersiz kılmıyor, iki farklı dağıtım modelini göstermek içindi)

## Neden her şey değil — kapsam kararı

Tek node'luk bir pratik ortamda (CRC) bazı şeyler **gerçekten** kurulup test edilebilir,
bazıları sadece "config'i doğru yazıp gerçek testi çok-node'lu prod'a bırakmak" anlamına
gelir. Bu ayrımı baştan netleştirdik:

| Konu | Bu klasörde durum |
|---|---|
| Custom SCC | ✅ Gerçek, canlı test edildi |
| TLS (cert-manager + autoGeneratedCerts) | ✅ Gerçek, canlı produce/consume testiyle kanıtlandı |
| SASL/PLAIN authentication | ✅ Gerçek, canlı testte kullanıldı |
| Resource requests/limits | ✅ Gerçek (ama pratik ortamın küçük boyutunda) |
| ArgoCD GitOps (pull-based sync) | ✅ Gerçek, canlı testle kanıtlandı |
| Pod anti-affinity | ⚠️ Config doğru yazıldı ama `preferred` (yumuşak) — tek node'da `required` kullansaydık pod'lar hiç açılmazdı. Gerçek multi-node'da AYNI YAML otomatik olarak sert izolasyon sağlar |
| Rack/multi-AZ awareness | ❌ Tek node'da test edilemez, bu klasöre eklenmedi |
| SASL/SCRAM | ❌ **CFK'nın kendisi SCRAM'i native desteklemiyor** (sadece configOverrides hack'i) — bu yüzden SASL/PLAIN kullandık, bu bir kısıtlama değil, doğrulanmış bir CFK gerçeği |
| Kurumsal internal PKI | ❌ Burada self-signed root CA kullanıldı — gerçek ortamda kurumun kendi CA'sından zincirlenmeli |
| Enterprise storage (ODF/Ceph/Trident) | ❌ Hâlâ CRC'nin hostpath-provisioner'ı |

## FAZ 1 — Custom SCC + namespace

Dosyalar: [`infra/namespace.yaml`](infra/namespace.yaml), [`infra/scc.yaml`](infra/scc.yaml)

```bash
oc apply -f prod-deployment/infra/namespace.yaml
oc apply -f prod-deployment/infra/scc.yaml
oc adm policy add-scc-to-user confluent-prod -z confluent-operator -n confluent-prod
oc adm policy add-scc-to-user confluent-prod -z default -n confluent-prod

helm upgrade --install confluent-operator confluentinc/confluent-for-kubernetes \
  --namespace confluent-prod \
  --set serviceAccount.create=true \
  --set serviceAccount.name=confluent-operator
```

**Playground'dan fark:** `--set podSecurity.enabled=false` **yok**. Onun yerine gerçek bir
`SecurityContextConstraints` objesi (UID 1000-1005, fsGroup 1001-1005 aralığında) tanımlayıp
sadece iki ServiceAccount'a (`confluent-operator`, `default`) `oc adm policy add-scc-to-user`
ile bağladık — CFK'nın varsayılan sabit UID 1001'i bu aralığın içinde kaldığı için hiçbir
`podSecurityContext` override'ına gerek kalmadı.

**Doğrulama:** `oc get pod kafka-0 -o jsonpath='{.metadata.annotations.openshift\.io/scc}'`
→ `confluent-prod` döner (`anyuid` veya `restricted-v2`'ye düşmediğinin kanıtı).

## FAZ 2 — cert-manager ile TLS

Dosyalar: [`cert-manager/00-namespace-and-operator.yaml`](cert-manager/00-namespace-and-operator.yaml),
[`cert-manager/10-root-ca.yaml`](cert-manager/10-root-ca.yaml)

Red Hat'in resmi **cert-manager Operator for Red Hat OpenShift**'i (`redhat-operators`
kataloğu, `stable-v1` kanalı) OLM üzerinden kuruldu — community Helm chart değil, bilerek
OpenShift-native/Red Hat destekli seçenek tercih edildi.

```bash
oc apply -f prod-deployment/cert-manager/00-namespace-and-operator.yaml
# CSV Succeeded olana kadar bekle, sonra:
oc apply -f prod-deployment/cert-manager/10-root-ca.yaml
```

**Mekanizma:** cert-manager sadece **root CA**'yı üretip `ca-pair-sslcerts` adlı secret'a
yazıyor (bu isim rastgele değil — CFK'nın `autoGeneratedCerts: true` özelliğinin aradığı
**tam isim**, Confluent dokümanından doğrulandı). Her broker/controller için **leaf**
sertifikaları CFK'nın kendisi üretip bu CA ile imzalıyor — cert-manager leaf sertifikalara
hiç karışmıyor, sadece kök otoritenin üretimi/rotasyonunu (`renewBefore: 720h`) yönetiyor.

**Dürüstlük notu:** Bu CA self-signed — kimse tarafından harici olarak güvenilmiyor, sadece
bizim cluster'ımız içinde geçerli. Gerçek kurumsal ortamda bu Issuer, kurumun internal
PKI'sinden zincirlenen bir Issuer ile değiştirilmeli.

## FAZ 3 — SASL/PLAIN credential'ları + prod Kafka/KRaft CR'ları

**Neden SCRAM değil PLAIN:** Confluent'ın resmi dokümanından doğrulandı — CFK, SASL/SCRAM'i
CRD API'sinde **native desteklemiyor** (sadece `configOverrides` ile manuel hack mümkün).
SASL/PLAIN ise `jaasConfig` ile temiz ve otomatik yönetiliyor. Bu bir kısıtlama tercihi değil,
doğrulanmış bir CFK gerçeği.

Credential secret'ları imperatif olarak oluşturuldu (asla git'e commit edilmedi):
```bash
oc create secret generic kraft-credential -n confluent-prod \
  --from-file=plain-users.json=... --from-file=plain-interbroker.txt=...
oc create secret generic kafka-credential -n confluent-prod \
  --from-file=plain-users.json=... --from-file=plain-interbroker.txt=...
```
- `kraft-credential`: KRaftController ve Kafka'nın `controllerListener` bölümü **aynı**
  secret'ı referans alır (Confluent dokümanının açıkça belirttiği bir kural — eşleşmezse
  Kafka, controller'a bağlanamaz).
- `kafka-credential`: Kafka'nın client-facing internal listener'ı için, ayrı bir "client"
  kullanıcısı içerir.

Prod CR'ları: [`kafka/kraft-controller.yaml`](kafka/kraft-controller.yaml),
[`kafka/kafka.yaml`](kafka/kafka.yaml). Playground'daki minimal CR'lardan farkları:
`tls.autoGeneratedCerts: true`, `listeners.*.tls.enabled + authentication.type: plain`,
`podTemplate.resources` (gerçek CPU/memory limitleri), `podTemplate.affinity.podAntiAffinity`
(`preferred`, tek-node kısıtı nedeniyle — yorum satırında gerekçesi yazılı), ve
**`podSecurityContext` YOK** (Custom SCC zaten hallediyor).

## FAZ 4 — ArgoCD (pull-based GitOps)

Dosyalar: [`argocd/00-gitops-operator.yaml`](argocd/00-gitops-operator.yaml),
[`argocd/05-rbac.yaml`](argocd/05-rbac.yaml), [`argocd/10-application.yaml`](argocd/10-application.yaml)

Red Hat'in resmi **OpenShift GitOps** operatörü kuruldu (`redhat-operators`, `latest`
kanalı — sadece `AllNamespaces` install modu destekleniyor, `openshift-operators`
namespace'ine kuruldu, otomatik olarak `openshift-gitops` namespace'i + hazır bir ArgoCD
instance'ı oluşturdu).

**Push (GitHub Actions) vs Pull (ArgoCD) — fark:**
- **Push**: Git'e push → GitHub event → runner tetiklenir → runner cluster'a bağlanıp
  `oc apply` çalıştırır. Cluster dışarıdan bir komutla değiştirilir, statik bir
  kubeconfig'in dışarı (GitHub Secret) çıkması gerekir.
- **Pull**: ArgoCD, cluster'ın **içinden** çalışan bir kontrolcü — sürekli git'i izler,
  git ile cluster arasındaki farkı (drift) kendisi bulup kendisi düzeltir. Dışarı hiçbir
  statik credential çıkmaz.

```bash
oc apply -f prod-deployment/argocd/00-gitops-operator.yaml
# CSV Succeeded olana kadar bekle

oc apply -f prod-deployment/argocd/05-rbac.yaml    # ArgoCD'ye confluent-prod'da izin ver
oc apply -f prod-deployment/argocd/10-application.yaml
```

**Application kapsam kararı:** `Application`, bilerek `prod-deployment/` kökü değil sadece
`prod-deployment/kafka/` alt klasörünü izliyor. `infra/`, `cert-manager/`, `argocd/`'nin
kendisi (operatör kurulumları) **bootstrap/platform** katmanı — gerçek kurumlarda bunları
platform ekibi bir kere elle kurar, her Kafka değişikliğinde tekrar dokunulmaz. Uygulama
(Kafka CR'ları) katmanını GitOps'a, altyapı katmanını manuel/admin sürecine ayırmak gerçek
kurumsal pratiktir.

`syncPolicy.automated`: `prune: true` (git'ten silinen, cluster'dan da silinir),
`selfHeal: true` (biri elle `oc edit` ile değiştirirse, ArgoCD otomatik geri döndürür).

## Doğrulama — gerçek testler

### 1) TLS + SASL/PLAIN uçtan uca test

CFK'nın ürettiği truststore ile gerçek bir authenticated+encrypted produce/consume yapıldı:

```bash
oc exec -n confluent-prod kafka-0 -- kafka-topics --bootstrap-server kafka:9071 \
  --command-config /tmp/client.properties --create --topic <test-topic> --partitions 3 --replication-factor 3
oc exec -n confluent-prod kafka-0 -- kafka-console-producer --bootstrap-server kafka:9071 \
  --topic <test-topic> --command-config /tmp/client.properties
oc exec -n confluent-prod kafka-0 -- kafka-console-consumer --bootstrap-server kafka:9071 \
  --topic <test-topic> --from-beginning --command-config /tmp/client.properties
```
**Sonuç:** Mesaj uçtan uca üretilip okundu (`Processed a total of 1 messages`). Ayrıca
kimlik doğrulaması **olmadan** bağlanma denemesi başarısız oldu (TLS-only porta çıplak
bağlanma girişimi bir `OutOfMemoryError` ile çöktü — temiz bir "access denied" değil ama
sonuç aynı: yetkisiz erişim çalışmıyor; kafka-0'ın asıl broker process'i bundan etkilenmedi,
`RESTARTS: 0` olarak doğrulandı).

### 2) ArgoCD GitOps canlı test

Kafka CR'ına git üzerinden bir annotation eklendi, **hiç `oc apply` çalıştırılmadan**:
```bash
git commit -am "..." && git push
oc annotate application kafka-prod -n openshift-gitops argocd.argoproj.io/refresh=hard --overwrite
oc get kafka kafka -n confluent-prod -o jsonpath='{.metadata.annotations.gitops-canli-test}'
```
**Sonuç:** Annotation cluster'daki gerçek objede göründü — git → ArgoCD → cluster zincirinin
insan müdahalesi olmadan çalıştığının kanıtı.

## Karşılaşılan sorunlar ve çözümleri (bu oturumda gerçekten yaşandı)

### 1. `jksPassword.txt` çıplak bir şifre değil, `key=value` formatında
CFK'nın ürettiği `kafka-generated-jks` secret'ındaki `jksPassword.txt` dosyasının içeriği
`jksPassword=<gerçek-şifre>` şeklinde — biz başta bu satırın **tamamını** şifre sanıp
kullandık, `keytool`/Kafka client'ı "Keystore was tampered with, or password was incorrect"
hatası verdi. Çözüm: `cut -d'=' -f2` ile `=` sonrasını parse etmek. **Bu tahmin edilerek
değil, dosyanın hex dump'ı incelenerek bulundu** — gerçek debug süreciydi.

### 2. `oc cp` çalışmadı: `tar` pod içinde yok
`cp-server` image'ı minimal, `tar` binary'si içermiyor, `oc cp` (içeride tar kullanır) bu
yüzden başarısız oldu. Çözüm: `oc exec -i ... -- sh -c 'cat > file' < local-file` ile
stdin üzerinden dosya aktarımı (tar gerektirmez, ayrıca şifre asla komut satırı argümanı
olarak görünmediği için daha güvenli).

### 3. `kafka-console-producer`/`kafka-topics --list` CLI sözdizimi değişmiş
Bu Kafka sürümünde `--broker-list` kaldırılmış, `--bootstrap-server` + `--command-config`
birleşik kalıbı kullanılıyor (`kafka-topics` ile aynı). Playground'daki eski örneklerden
kopyalanan komutlar bu yüzden hata verdi — TLS/SASL ile ilgisi yoktu, sadece CLI sürüm farkı.

### 4. Tek node'da ArgoCD'nin varsayılan kaynak istekleri sığmadı
OpenShift'in kendi sistem pod'ları (etcd, apiserver, ingress vb.) tek node'da zaten ~2.4 CPU
tüketiyor. `confluent-prod` (2.5 CPU) + ArgoCD'nin varsayılan istekleri (her bileşen 250m,
+dex) toplamı node'un 5.8 CPU'luk kapasitesini aştı → `Insufficient cpu`, birden fazla
ArgoCD pod'u `Pending` kaldı. Çözüm: `ArgoCD` CR'ını patch'leyip her bileşenin isteğini
düşürdük, kullanmadığımız `dex`/SSO'yu tamamen kapattık (`spec.sso: null`).

Bunun **yan etkisi** ilginçti: kaynak değişikliği yeni bir rolling update tetikledi, ama
node zaten dolu olduğu için yeni (küçük) pod'lar da schedule olamadı, eski pod'lar da
"yenisi hazır olana kadar" silinmeyi reddetti — klasik bir **kilitlenme**. Çözüm: eski
ReplicaSet'leri doğrudan `oc scale rs ... --replicas=0` ile sıfıra çekmek (Deployment'ın
kendi rollout temposunu bypass etmek).

**Bu, pratik ortama özgü bir sorun** — gerçek prod node'larında bu kadar sıkışık kaynak
olmayacağı için muhtemelen hiç yaşanmaz, ama "kaynak sıkışıklığında rolling update
kilitlenmesi" genel bir Kubernetes bilgisi olarak değerli.

### 5. ArgoCD, Kafka CR'larını patch edemedi: RBAC eksik
`system:serviceaccount:openshift-gitops:openshift-gitops-argocd-application-controller`
kimliğinin `confluent-prod` namespace'inde `kafkas`/`kraftcontrollers` üzerinde hiç yetkisi
yoktu (`Forbidden`). Çözüm: GitHub Actions için yazdığımız least-privilege RBAC
deseninin birebir aynısını (bkz. [`argocd/05-rbac.yaml`](argocd/05-rbac.yaml)) ArgoCD'nin
kimliğine de tanımladık.

### 6. ArgoCD "app path does not exist" hatası verdi
En basit ama en öğretici hata: `prod-deployment/` klasörünü **hiç git'e commit/push
etmemiştik** — tüm bu çalışma sadece yerel diskteydi. ArgoCD gerçek GitHub içeriğine bakıyor,
orada klasör yoktu. Çözüm: `git add/commit/push`. **GitOps'un en temel kuralı**: git = tek
gerçek kaynak, orada olmayan bir şey cluster'a asla yansımaz.
