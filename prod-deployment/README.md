# Prod-Shaped Kafka (KRaft) Deployment — Custom SCC + TLS + SASL + GitOps

This folder is the **security- and operations-hardened** version of the playground setup
in the main [README](../README.md). Three things that were deliberately skipped/simplified
in the playground were built here with real mechanisms and **proven with live tests**:

1. **Custom SCC** (instead of the playground's `podSecurity.enabled=false`)
2. **TLS + SASL/PLAIN** (instead of the playground's plaintext PLAINTEXT)
3. **ArgoCD (pull-based GitOps)** (instead of the playground's GitHub Actions/push-based —
   the two aren't mutually exclusive, this was meant to demonstrate two different
   deployment models)

## Why not everything — scoping decision

In a single-node practice environment (CRC), some things can **genuinely** be built and
tested, while others only mean "write the config correctly and leave the real test to a
multi-node prod." We drew this line up front:

| Topic | Status in this folder |
|---|---|
| Custom SCC | ✅ Real, live-tested |
| TLS (cert-manager + autoGeneratedCerts) | ✅ Real, proven with a live produce/consume test |
| SASL/PLAIN authentication | ✅ Real, used in the live test |
| Resource requests/limits | ✅ Real (but sized for the small practice environment) |
| ArgoCD GitOps (pull-based sync) | ✅ Real, proven with a live test |
| Pod anti-affinity | ⚠️ Config written correctly but `preferred` (soft) — `required` would have left pods stuck on a single node. On a real multi-node cluster the SAME YAML enforces hard isolation automatically |
| Rack/multi-AZ awareness | ❌ Can't be tested on a single node, not added to this folder |
| SASL/SCRAM | ❌ **CFK itself doesn't natively support SCRAM** (only a configOverrides hack) — that's why we used SASL/PLAIN; this isn't a limitation we chose, it's a verified CFK fact |
| Enterprise internal PKI | ❌ A self-signed root CA was used here — in a real environment this should be chained from the organization's own CA |
| Enterprise storage (ODF/Ceph/Trident) | ❌ Still CRC's hostpath-provisioner |

## Architecture

```
                    ┌─────────────────────────┐
                    │   Git (main branch)      │
                    │   prod-deployment/kafka/  │
                    └────────────┬─────────────┘
                                 │ ArgoCD continuously watches (pull)
                                 ▼
                    ┌─────────────────────────┐
                    │   ArgoCD Application      │  (openshift-gitops ns)
                    │   'kafka'                 │  selfHeal + auto-prune
                    └────────────┬─────────────┘
                                 │ apply (scoped by RBAC)
                                 ▼
        ┌────────────────────────────────────────────┐
        │              confluent namespace              │
        │  ┌──────────────┐      ┌──────────────────┐ │
        │  │ KRaftController│◄────┤   Kafka (3 brokers)│ │
        │  │  (3 pods, TLS) │TLS  │  (TLS+SASL/PLAIN)  │ │
        │  └──────────────┘      └──────────────────┘ │
        │         Under Custom SCC (UID 1000-1005)       │
        │         Certificates signed by the cert-manager │
        │         CA (ca-pair-sslcerts)                   │
        └────────────────────────────────────────────┘
```

## PHASE 1 — Custom SCC + namespace

Files: [`infra/namespace.yaml`](infra/namespace.yaml), [`infra/scc.yaml`](infra/scc.yaml)

```bash
oc apply -f prod-deployment/infra/namespace.yaml
oc apply -f prod-deployment/infra/scc.yaml
oc adm policy add-scc-to-user confluent-scc -z confluent-operator -n confluent
oc adm policy add-scc-to-user confluent-scc -z default -n confluent

helm upgrade --install confluent-operator confluentinc/confluent-for-kubernetes \
  --namespace confluent \
  --set serviceAccount.create=true \
  --set serviceAccount.name=confluent-operator
```

**Difference from the playground:** `--set podSecurity.enabled=false` is **absent**.
Instead, we defined a real `SecurityContextConstraints` object (UID 1000-1005, fsGroup
1001-1005 range) and bound it to just two ServiceAccounts (`confluent-operator`, `default`)
via `oc adm policy add-scc-to-user` — since CFK's default fixed UID 1001 falls within this
range, no `podSecurityContext` override was needed at all.

**Verification:** `oc get pod kafka-0 -o jsonpath='{.metadata.annotations.openshift\.io/scc}'`
→ returns `confluent-scc` (proof it didn't fall back to `anyuid` or `restricted-v2`).

## PHASE 2 — TLS with cert-manager

Files: [`cert-manager/00-namespace-and-operator.yaml`](cert-manager/00-namespace-and-operator.yaml),
[`cert-manager/10-root-ca.yaml`](cert-manager/10-root-ca.yaml)

Red Hat's official **cert-manager Operator for Red Hat OpenShift** (`redhat-operators`
catalog, `stable-v1` channel) was installed via OLM — not the community Helm chart; the
OpenShift-native/Red Hat-supported option was deliberately chosen.

```bash
oc apply -f prod-deployment/cert-manager/00-namespace-and-operator.yaml
# Wait for the CSV to reach Succeeded, then:
oc apply -f prod-deployment/cert-manager/10-root-ca.yaml
```

**Mechanism:** cert-manager only generates the **root CA** and writes it to a secret named
`ca-pair-sslcerts` (this name isn't arbitrary — it's the **exact name** CFK's
`autoGeneratedCerts: true` feature looks for, confirmed from the Confluent documentation).
For each broker/controller, CFK itself generates and signs the **leaf** certificates with
this CA — cert-manager never touches leaf certificates, it only manages the root
authority's generation/rotation (`renewBefore: 720h`).

**Honesty note:** This CA is self-signed — it isn't trusted externally by anyone, it's only
valid within our own cluster. In a real enterprise environment, this Issuer should be
replaced with one chained from the organization's internal PKI.

## PHASE 3 — SASL/PLAIN credentials + production Kafka/KRaft CRs

**Why PLAIN, not SCRAM:** confirmed from Confluent's official documentation — CFK does
**not natively support** SASL/SCRAM in the CRD API (only a manual `configOverrides` hack is
possible). SASL/PLAIN, on the other hand, is cleanly and automatically managed via
`jaasConfig`. This isn't a preference, it's a verified CFK fact.

Credential secrets were created imperatively (never committed to git):
```bash
oc create secret generic kraft-credential -n confluent \
  --from-file=plain-users.json=... --from-file=plain-interbroker.txt=...
oc create secret generic kafka-credential -n confluent \
  --from-file=plain-users.json=... --from-file=plain-interbroker.txt=...
```
- `kraft-credential`: referenced **identically** by both KRaftController and Kafka's
  `controllerListener` section (a rule Confluent's documentation states explicitly —
  otherwise Kafka can't connect to the controller).
- `kafka-credential`: for Kafka's client-facing internal listener, contains a separate
  `client` user.

Production CRs: [`kafka/kraft-controller.yaml`](kafka/kraft-controller.yaml),
[`kafka/kafka.yaml`](kafka/kafka.yaml). Differences from the playground's minimal CRs:
`tls.autoGeneratedCerts: true`, `listeners.*.tls.enabled + authentication.type: plain`,
`podTemplate.resources` (real CPU/memory limits), `podTemplate.affinity.podAntiAffinity`
(`preferred`, due to the single-node constraint — the reasoning is written in a comment),
and **no `podSecurityContext`** (the Custom SCC already handles it).

## PHASE 4 — ArgoCD (pull-based GitOps)

Files: [`argocd/00-gitops-operator.yaml`](argocd/00-gitops-operator.yaml),
[`argocd/05-rbac.yaml`](argocd/05-rbac.yaml), [`argocd/10-application.yaml`](argocd/10-application.yaml)

Red Hat's official **OpenShift GitOps** operator was installed (`redhat-operators`,
`latest` channel — only the `AllNamespaces` install mode is supported, so it was installed
into `openshift-operators` namespace, which automatically created the `openshift-gitops`
namespace + a ready ArgoCD instance).

**Push (GitHub Actions) vs Pull (ArgoCD) — the difference:**
- **Push**: push to git → GitHub event → runner triggered → the runner connects to the
  cluster and runs `oc apply`. The cluster is changed by an external command, requiring a
  static kubeconfig to leave the cluster (as a GitHub Secret).
- **Pull**: ArgoCD is a controller running **inside** the cluster — it continuously watches
  git, finds the drift between git and the cluster, and fixes it itself. No static
  credential ever leaves the cluster.

```bash
oc apply -f prod-deployment/argocd/00-gitops-operator.yaml
# Wait for the CSV to reach Succeeded

oc apply -f prod-deployment/argocd/05-rbac.yaml    # grant ArgoCD permission in confluent
oc apply -f prod-deployment/argocd/10-application.yaml
```

**Application scoping decision:** the `Application` deliberately watches only the
`prod-deployment/kafka/` subfolder, not the `prod-deployment/` root. `infra/`,
`cert-manager/`, and `argocd/` itself (the operator installations) are the
**bootstrap/platform** layer — in real organizations these are installed once, manually, by
the platform team, and are not touched again on every Kafka change. Separating the
application layer (Kafka CRs) into GitOps and the infrastructure layer into a
manual/admin process is genuine enterprise practice.

`syncPolicy.automated`: `prune: true` (a resource removed from git is also removed from the
cluster), `selfHeal: true` (if someone changes something manually with `oc edit`, ArgoCD
automatically reverts it).

## Verification — real tests

### 1) End-to-end TLS + SASL/PLAIN test

A real authenticated+encrypted produce/consume was performed using the truststore CFK
generated:

```bash
oc exec -n confluent kafka-0 -- kafka-topics --bootstrap-server kafka:9071 \
  --command-config /tmp/client.properties --create --topic <test-topic> --partitions 3 --replication-factor 3
oc exec -n confluent kafka-0 -- kafka-console-producer --bootstrap-server kafka:9071 \
  --topic <test-topic> --command-config /tmp/client.properties
oc exec -n confluent kafka-0 -- kafka-console-consumer --bootstrap-server kafka:9071 \
  --topic <test-topic> --from-beginning --command-config /tmp/client.properties
```
**Result:** a message was produced and consumed end-to-end (`Processed a total of 1
messages`). In addition, attempting to connect **without** authentication failed (a bare
connection attempt to a TLS-only port crashed with an `OutOfMemoryError` — not a clean
"access denied," but the outcome is the same: unauthorized access does not work; kafka-0's
actual broker process was unaffected by this, confirmed via `RESTARTS: 0`).

### 2) ArgoCD GitOps live test

An annotation was added to the Kafka CR via git, **without ever running `oc apply`**:
```bash
git commit -am "..." && git push
oc annotate application kafka -n openshift-gitops argocd.argoproj.io/refresh=hard --overwrite
oc get kafka kafka -n confluent -o jsonpath='{.metadata.annotations.gitops-live-test}'
```
**Result:** the annotation appeared on the real object in the cluster — proof that the
git → ArgoCD → cluster chain works without any human intervention.

## Problems encountered and how they were solved (all genuinely occurred in this session)

### 1. `jksPassword.txt` is not a bare password, it's in `key=value` format
The `jksPassword.txt` file inside the `kafka-generated-jks` secret CFK generates contains
`jksPassword=<actual-password>` — at first we treated the **entire** line as the password,
and `keytool`/the Kafka client threw "Keystore was tampered with, or password was
incorrect." Fix: parse everything after `=` with `cut -d'=' -f2`. **This was found not by
guessing but by inspecting a hex dump of the file** — a genuine debugging process.

### 2. `oc cp` didn't work: no `tar` inside the pod
The `cp-server` image is minimal and doesn't include the `tar` binary, so `oc cp` (which
uses tar internally) failed. Fix: transfer the file via stdin with
`oc exec -i ... -- sh -c 'cat > file' < local-file` (doesn't require tar, and is also more
secure since the password never appears as a command-line argument).

### 3. `kafka-console-producer`/`kafka-topics --list` CLI syntax had changed
This Kafka version has removed `--broker-list`; it uses the combined
`--bootstrap-server` + `--command-config` pattern (same as `kafka-topics`). Commands copied
from the older playground examples failed for this reason — it had nothing to do with
TLS/SASL, just a CLI version difference.

### 4. ArgoCD's default resource requests didn't fit on a single node
OpenShift's own system pods (etcd, apiserver, ingress, etc.) already consume ~2.4 CPU on a
single node. `confluent` (2.5 CPU) + ArgoCD's default requests (250m per component, +dex)
together exceeded the node's 5.8 CPU capacity → `Insufficient cpu`, several ArgoCD pods
stayed `Pending`. Fix: patched the `ArgoCD` CR to lower each component's requests, and
disabled the `dex`/SSO we weren't using entirely (`spec.sso: null`).

The **side effect** of this was interesting: the resource change triggered a new rolling
update, but since the node was already full, the new (smaller) pods couldn't schedule
either, and the old pods refused to be deleted "until the replacement is ready" — a classic
**deadlock**. Fix: scale the old ReplicaSets directly to zero with
`oc scale rs ... --replicas=0` (bypassing the Deployment's own rollout pacing).

**This is specific to the practice environment** — it likely won't happen on real
production nodes since they won't be this resource-constrained, but "rolling update
deadlock under resource pressure" is valuable general Kubernetes knowledge.

### 5. ArgoCD couldn't patch the Kafka CRs: missing RBAC
The identity
`system:serviceaccount:openshift-gitops:openshift-gitops-argocd-application-controller`
had no permissions at all on `kafkas`/`kraftcontrollers` in the `confluent` namespace
(`Forbidden`). Fix: defined the exact same least-privilege RBAC pattern we wrote for GitHub
Actions (see [`argocd/05-rbac.yaml`](argocd/05-rbac.yaml)) for ArgoCD's identity as well.

### 6. ArgoCD threw an "app path does not exist" error
The simplest but most instructive error: we had **never committed/pushed** the
`prod-deployment/` folder to git — all of this work only existed on local disk. ArgoCD
looks at the actual GitHub content, and the folder wasn't there. Fix: `git add/commit/push`.
**GitOps's most fundamental rule**: git is the single source of truth — anything not there
never reaches the cluster.
